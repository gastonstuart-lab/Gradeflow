# Product Expansion Plan

## Recommended Product Tiers

### Free

Target user: individual teacher evaluating the product.

Include:

- Class and student roster management
- Basic gradebook workflows
- Seating plans
- Basic exports
- Timetable viewing
- Manual reminders and calendar utilities
- Limited imports

Purpose:
show clear daily value fast, create habit, and make GradeFlow easy to recommend peer-to-peer.

### Pro Teacher

Target user: serious individual teacher who wants time savings and reliability.

Include everything in Free, plus:

- Advanced import flows for messy school files
- AI-assisted import analysis
- Saved templates and reusable grading structures
- Richer exports and result packaging
- More classroom tools and presentation workflows
- Cloud sync for core teacher workspace data
- Better backup / restore confidence

Purpose:
this is the first real paid tier and should monetize time saved, reduced admin friction, and peace of mind.

### Department

Target user: subject lead or small department coordinating multiple teachers.

Include everything in Pro, plus:

- Shared grading templates
- Shared roster / class import pipelines
- Shared seating / room setup libraries
- Department calendars and planning views
- Role-aware collaboration basics
- Shared export standards

Purpose:
turn GradeFlow from a personal efficiency tool into a team workflow product.

### School

Target user: school leadership, admin teams, or IT-backed rollout.

Include everything in Department, plus:

- School-level configuration and branding
- Admin onboarding and bulk imports
- Role and permission controls
- Shared data boundaries and auditability
- Centralized sync confidence
- Supportable deployment / recovery model

Purpose:
make the product sellable as an institutional workflow layer, not just a powerful teacher app.

## Strongest Current Commercial Features

The most defensible current value is in the operational core:

1. Roster management and class organization
2. Gradebook and final results workflows
3. Exports that help teachers finish required admin work
4. Seating plans and classroom execution tools
5. Schedule / timetable imports from real school documents
6. Flexibility around messy file formats

These features map directly to teacher pain and time saved, which is the best starting point for monetization.

## What Most Increases Willingness To Pay

Highest-leverage future additions:

1. Reliable cloud-backed source of truth for core records
2. Shared grading templates across teachers
3. Shared imports and department-level setup reuse
4. Faster onboarding from school spreadsheets with fewer manual fixes
5. Better admin-grade exports and report packaging
6. Multi-device confidence and backup reassurance
7. Lightweight permissions for heads of department / coordinators

The underlying pattern:
teachers pay for saved time and reduced risk; schools pay for standardization, rollout, and control.

## Staged Monetization Roadmap

### Stage 1: Solo Teacher Value

Focus on:

- polish grading, roster, export, seating, timetable, and imports
- remove ambiguity around storage and reliability
- make the dashboard feel like a daily teacher command center

Commercial motion:

- free evaluation tier
- paid Pro teacher tier

### Stage 2: Trusted Teacher Workspace

Focus on:

- stronger cloud sync for core records
- reusable templates
- higher confidence import and export flows
- usage analytics and product feedback loops

Commercial motion:

- upsell active users from Free to Pro
- start building case studies around time saved

### Stage 3: Department Workflow

Focus on:

- shared templates
- shared room / seating assets
- shared planning surfaces
- role-aware collaboration boundaries

Commercial motion:

- department pricing
- pilot one subject department at a time

### Stage 4: School Rollout

Focus on:

- admin configuration
- school onboarding
- permissions
- supportability and auditability

Commercial motion:

- annual contracts
- onboarding / migration services if needed

## What Needs To Be True Technically Before Charging Schools

Minimum technical bar:

1. Core academic data must have a clearly documented source of truth.
2. Core teacher records must sync reliably across devices when cloud mode is enabled.
3. School-specific defaults must live in configuration, not scattered hardcoded strings.
4. Shared versus personal data boundaries must be explicit.
5. Imports must be robust enough for real school spreadsheets without founder intervention.
6. Exports must be stable and reproducible.
7. Permissions and ownership rules must have a plausible extension path.
8. The app must have basic operational confidence:
   error reporting, backup strategy, migration strategy, and safe upgrade paths.

## Recommended Commercial Positioning

GradeFlow should be positioned first as:

`the teacher operating system that reduces daily admin drag`

Then expanded into:

`the department and school workflow layer that standardizes grading, roster, planning, and classroom operations`

That sequence matches the codebase’s current strengths and avoids premature enterprise theater.
